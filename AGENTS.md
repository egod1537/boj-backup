# BOJ Backup CLI — 명세서 (v0.2)

## 1. 개요

백준 온라인 저지(BOJ)의 제출 코드, 문제 본문, 메타데이터를 자동으로 수집하여 Git 레포에 백업하는 CLI 도구.

## 2. 목표

- **원클릭 백업**: 한 번의 명령으로 전체 제출 히스토리를 Git 레포에 동기화
- **증분 동기화**: 이미 백업된 제출은 건너뛰고 새 제출만 추가
- **오프라인 열람**: 문제 본문 + 코드가 함께 저장되어 네트워크 없이도 풀이 히스토리 탐색 가능

## 3. 데이터 소스

| 데이터 | 소스 | 방법 |
|---|---|---|
| 제출 목록 | `acmicpc.net/status?user_id={id}` | HTML 스크래핑 (페이지네이션) |
| 소스 코드 | `acmicpc.net/source/{submission_id}` | HTML 스크래핑 (로그인 필요) |
| 문제 본문 | `acmicpc.net/problem/{problem_id}` | HTML 스크래핑 → Markdown 변환 |
| 문제 메타 | `solved.ac/api/v3/problem/show?problemId={id}` | Public REST API (인증 불필요) |

### 3.1 인증

- `.env` 파일에 `BOJ_ID`와 `BOJ_PW`를 저장
- CLI가 직접 BOJ 로그인 수행 → 세션 쿠키 획득 후 메모리에 유지
- 세션 만료 시 자동 재로그인 시도

```env
BOJ_ID=myhandle
BOJ_PW=mypassword
```

> `.env`는 `.gitignore`에 자동 추가하여 레포에 커밋되지 않도록 보장

## 4. 백업 대상 및 필터

### 4.1 기본 대상

- **모든 제출** (AC, WA, TLE, RE, CE 등 결과 무관)
- 같은 문제에 대한 모든 제출을 전부 보존 (중복 제거 없음)
- 각 제출의 결과(AC/WA/TLE 등)는 파일 헤더 및 메타데이터로 기록

### 4.2 선택적 필터

| 플래그 | 설명 | 기본값 |
|---|---|---|
| `--ac-only` | AC 제출만 가져오기 | `false` |
| `--problem-range` | 특정 문제 번호 범위만 (e.g. `1000-2000`) | 전체 |
| `--after` | 특정 날짜 이후 제출만 | 없음 |
| `--language` | 특정 언어만 (e.g. `C++17`) | 전체 |

**`{problem_id}/README.md`** — 문제 본문

```markdown
|---|---|
| 티어 | Bronze V |
| 태그 | math, implementation |

## 문제
두 정수 A와 B를 입력받은 다음, A+B를 출력하는 프로그램을 작성하시오.

## 입력
...

## 출력
...
```

**`{problem_id}/meta.json`** — 구조화된 메타데이터
```json
{
  "problemId": 1000,
  "titleKo": "A+B",
  "level": 1,
  "tags": ["math", "implementation"],
  "timeLimit": "2s",
  "memoryLimit": "128MB",

```

```cpp
// BOJ 1000 - A+B | AC | 2024-03-15 | 0ms | 1234KB | C++17
#include <bits/stdc++.h>

```

## 6. CLI 인터페이스

```
boj-backup <command> [options]
```

### 6.1 Commands

| 커맨드 | 설명 |
|---|---|
| `init` | 현재 디렉토리를 boj-backup 레포로 초기화 (git init + config 생성) |
| `sync` | 새 제출을 가져와서 커밋 (증분 동기화) |
| `sync --full` | 전체 히스토리를 처음부터 재동기화 |
| `push` | 리모트 레포에 push |
| `config` | 설정 보기/수정 (리모트 URL, 커밋 스타일 등) |
| `stats` | 백업 통계 출력 (문제 수, 제출 수, 티어 분포) |
| `tui` | 인터랙티브 TUI 모드 진입 (동기화 모니터링 + 설정 관리) |
| `serve` | 로컬 웹 뷰어 실행 (`--port`, `--open` 옵션) |

### 6.2 설정 파일 (`boj-backup.json`)

```json
{
  "remote": "git@github.com:user/boj-backup.git",
  "commitStyle": "per-problem",
  "includeHeader": true
}
```

> 인증 정보(`BOJ_ID`, `BOJ_PW`)는 `.env`에 별도 관리. 설정 파일에는 포함하지 않음.

### 6.3 커밋 전략

| 옵션 | 동작 |
|---|---|
| `per-problem` | 문제 단위 커밋: `[BOJ 1000] A+B (Bronze V, C++17)` |
| `per-sync` | 동기화 단위 커밋: `sync: 15 new submissions (2026-04-16)` |
| `per-submission` | 제출 단위 커밋 (커밋 시각 = 제출 시각으로 backdating) |

**기능:**

- 리모트 연결 테스트 (git ls-remote)
- 저장 시 `.env` + `boj-backup.json` 동시 업데이트

## 8. Rate Limiting & 예의

- BOJ 요청 간 **기본 3초 딜레이** (`--delay` 플래그로 조절 가능, 최솟값 2초)
- 연속 요청 시 **랜덤 지터** 추가 (±0.5초) → 기계적 패턴 회피
- 429/403 응답 시 **지수 백오프**: 10초 → 30초 → 60초 → 중단
- solved.ac API는 별도 rate limit 준수 (기본 1초 딜레이)
- 동시 요청 금지 (모든 요청 직렬 처리)
- User-Agent 헤더에 도구명 + 버전 명시 (`boj-backup/0.1.0`)
- 실패 시 지수 백오프 재시도 (최대 3회)
- 대량 동기화(500+ 제출) 시 50건마다 **30초 쿨다운** 자동 삽입

## 9. 체크포인트 & 재개 (Checkpoint/Resume)

크롤링은 수백~수천 건의 요청이 필요하므로 중간에 끊길 가능성이 높다. BOJ 제출 번호는 **단조 증가하는 정수**이므로, 이를 자연스러운 커서로 활용한다.

### 9.1 핵심 아이디어

- 제출 번호(submission ID)는 전체 BOJ에서 글로벌하게 증가
- "여기까지 처리 완료" 한 줄로 진행 상태를 표현 가능
- 증분 동기화 시 마지막 처리 ID 이후의 제출만 가져오면 됨

### 9.2 체크포인트 파일 (`.boj-checkpoint.json`)

동기화 진행 중 실시간으로 갱신되는 로컬 상태 파일. `.gitignore`에 포함.

```json
{
  "startedAt": "2026-04-16T14:30:00Z",
  "phase": "fetch-source",
  "submissions": {
    "discovered": [67891230, 67891231, 67891232, "..."],
    "cursor": 67891231
  },
  "problems": {
    "discovered": [1000, 1753, 2042, "..."],
    "cursor": 1753
  },
  "errors": [
    { "submissionId": 67891236, "phase": "source", "error": "403", "retries": 3 }
  ]
}
```

| 필드 | 의미 |
|---|---|
| `submissions.discovered` | Phase 1에서 수집한 전체 제출 ID 목록 (정렬됨) |
| `submissions.cursor` | Phase 2에서 마지막으로 소스코드 fetch 완료한 제출 ID |
| `problems.discovered` | 제출 목록에서 추출한 고유 문제 ID 목록 |
| `problems.cursor` | Phase 3에서 마지막으로 문제 본문 fetch 완료한 문제 ID |
| `errors` | 3회 재시도 후에도 실패한 항목 기록 |

### 9.3 동기화 파이프라인 (3-Phase)

```
Phase 1: 제출 목록 수집          Phase 2: 소스코드 fetch         Phase 3: 문제 본문/메타 fetch
─────────────────────           ─────────────────────          ──────────────────────────
/status 페이지네이션 순회    →    discovered 리스트를             discovered 리스트를
제출 ID 전부 수집                cursor 다음부터 순차 요청   →    cursor 다음부터 순차 요청
                                 (건별 cursor 갱신)              (건별 cursor 갱신)
```

- Phase 1 완료 → `submissions.discovered` 확정, Phase 2 시작
- Phase 2 진행 중 → 소스코드 하나 받을 때마다 `submissions.cursor` 갱신
- Phase 3 진행 중 → 문제 하나 받을 때마다 `problems.cursor` 갱신
- `sync` 재실행 시 체크포인트가 있으면 cursor 위치부터 자동 재개

### 9.4 재개 동작

| 상황 | 동작 |
|---|---|
| Phase 1 중 중단 | discovered가 비어있으므로 Phase 1부터 재시작 |
| Phase 2 중 중단 | `submissions.cursor` 다음 ID부터 재개 |
| Phase 3 중 중단 | `problems.cursor` 다음 ID부터 재개 |
| 정상 완료 | 체크포인트 삭제, `meta.json`에 마지막 제출 ID + 시각 기록 |
| 에러 스킵 항목 | `errors`에 기록, 완료 후 요약 출력 |

### 9.5 증분 동기화와의 연계

`meta.json`에 마지막 성공 동기화의 최대 제출 ID를 기록:

```json
{
  "lastSyncedSubmissionId": 67891237,
  "lastSyncedAt": "2026-04-16T15:45:00Z"
}
```

다음 `sync` 실행 시 Phase 1에서 이 ID보다 큰 제출만 수집 → 자연스럽게 증분 동기화.

### 9.6 CLI / TUI 표시

## 10. 에러 처리

| 상황 | 동작 |
|---|---|
| 세션 만료 | `.env` 자격증명으로 자동 재로그인 시도, 실패 시 중단 |
| 네트워크 오류 | 재시도 3회 후 해당 제출 스킵, 로그 기록 |
| BOJ 403/429 | 딜레이 증가 후 재시도 |
| 파싱 실패 | 해당 문제 스킵 + 경고 로그 |
| Git 충돌 | force push 옵션 또는 수동 해결 안내 |

## 11. 웹 프론트엔드 (로컬 뷰어)

백업된 데이터를 BOJ와 유사한 UI로 브라우징할 수 있는 로컬 웹 서버. `boj-backup serve`로 실행.

### 11.1 기술 스택

| 구성 | 선택 | 비고 |
|---|---|---|
| 프레임워크 | **Vite + React** | HMR, 빠른 빌드 |
| 스타일링 | `tailwindcss` | BOJ 스타일 재현 용이 |
| 코드 하이라이팅 | `shiki` 또는 `prism-react-renderer` | 다국어 구문 강조 |
| Markdown 렌더링 | `react-markdown` + `remark-math` + `rehype-katex` | 수식 포함 문제 본문 |
| 라우팅 | `react-router` | SPA 내비게이션 |
| 데이터 소스 | 로컬 파일시스템 (Express 미들웨어 또는 Vite plugin) | meta.json + README.md 직접 읽기 |

- 페이지네이션 또는 무한 스크롤

#### 문제 상세 (`/problem/:id`)

```
│  ## 입력                                                    │
│  첫째 줄에 정점의 개수 V와 간선의 개수 E가...               │
```

**기능:**
- 문제 본문 Markdown 렌더링 (수식 KaTeX 지원)
- 예제 입출력 복사 버튼
- 내 제출 목록 (결과별 색상 구분: AC=초록, WA=빨강, TLE=노랑)
- 코드 접기/펼치기 (인라인 코드 뷰어)
- BOJ 원문 링크 (`acmicpc.net/problem/:id`)

### 11.3 데이터 플로우

```
boj-backup 레포 (로컬 파일시스템)
    │
    ├── */meta.json ──────┐
    ├── */README.md ──────┤
    └── */submissions/ ───┤
                          ▼
              Vite dev server
              (Express 미들웨어)
                    │
                    ▼
              /api/problems      → 문제 목록 + 메타 (검색/필터용)
              /api/problem/:id   → 문제 본문 + 제출 목록
              /api/submission/:id → 소스코드 원문
              /api/stats          → 대시보드 통계 (집계)
                    │
                    ▼
              React SPA (Vite HMR)
```

- 서버 시작 시 전체 `meta.json`을 메모리에 인덱싱 (검색/필터 성능)
- Markdown 렌더링은 클라이언트 사이드 (react-markdown)
- 코드 하이라이팅도 클라이언트 사이드 (shiki wasm)

### 11.4 CLI 연동

```bash
boj-backup serve              # localhost:3000에서 뷰어 실행
boj-backup serve --port 8080  # 포트 지정
boj-backup serve --open       # 브라우저 자동 열기
```

`sync` 후 자동으로 인덱스 갱신. 뷰어 실행 중 `sync`를 별도 터미널에서 돌리면 새로고침으로 반영.
